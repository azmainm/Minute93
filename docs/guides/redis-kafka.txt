What is Docker and why do you need it?
Your app needs three things running alongside it: a database (Postgres), a fast cache (Redis), and a message queue (Redpanda/Kafka). These are separate programs — they're not part of your NestJS code. They need to be installed, configured, and running on your machine before your app can talk to them.
Without Docker, you'd have to install Postgres, install Redis, install Redpanda — each one separately, each with its own configuration, each potentially conflicting with other stuff on your laptop. And if your colleague (or your future self on a different machine) wants to run the project, they'd have to do the same manual setup.
Docker solves this. Think of Docker as a box factory. Each "container" is a sealed box that has one program inside it, pre-configured and ready to go. Your docker-compose.yml file says "I need three boxes: one with Postgres, one with Redis, one with Redpanda." You type docker compose up and all three boxes start running. Type docker compose down and they all stop. No installation, no configuration conflicts, nothing polluting your laptop. Delete the boxes anytime, recreate them in seconds.
Your NestJS app runs on your laptop normally (not in Docker) and talks to these three boxes via localhost — same as if you'd installed them natively, but with zero setup pain.
How Redis works in your project
Redis is an extremely fast key-value store that lives entirely in memory (RAM). Think of it as a giant JavaScript object ({}) that any part of your system can read from and write to, and it responds in under 1 millisecond.
Your app uses it for four things:
Caching: When someone loads the Arsenal vs Chelsea match page, NestJS first asks Redis: "do you have the score for match 42?" If yes, Redis returns it instantly. If no, NestJS queries Postgres (slower), gets the answer, saves it to Redis for next time, then responds. This means the first person to ask pays the "slow" cost, and everyone after gets the fast cached version.
Rate limiting: Every time a user hits your API, NestJS increments a counter in Redis: "user 123 has made 15 requests in the last minute." If it exceeds the limit, the request is blocked. Redis is perfect for this because it can automatically expire these counters after 60 seconds.
Pub/Sub: When a goal happens, a Kafka consumer needs to tell ALL connected browsers about it. It "publishes" the event to a Redis channel. Every NestJS instance that has browsers connected is "subscribed" to that channel. Redis delivers the message to all subscribers instantly.
Deduplication: The poller stores IDs of events it has already processed. Before processing a new event, it checks: "is this ID already in the set?" If yes, skip it. Redis Sets are perfect for this — instant lookups.
Redis locally vs production
Locally: A Docker container running Redis on your laptop. Your app connects to localhost:6379. Free, instant, works offline.
Production: Render's managed Redis (they call it "Key Value"). It's the same Redis, but Render runs it for you on their servers. Your NestJS app on Render connects to it via Render's internal private network — meaning they're in the same data center, the connection never leaves Render's network, and latency is sub-millisecond. This is why you use Render's Redis instead of an external service like Upstash — an external service would add 10-50ms of internet round-trip to every Redis call, which defeats the purpose of a cache that's supposed to be fast.
You don't need a third party because Render already provides this.
How Kafka (Redpanda) works in your project
Kafka is a message queue — a conveyor belt for data. One part of your system puts messages on the belt (the "producer"), and other parts take messages off the belt (the "consumers"). The key feature is: multiple consumers can each independently read the same message. Put one message on, four different workers each get a copy.
In your app, the Poller is the producer. Every 30 seconds it asks API-Football what's happening, and when it detects a new event (a goal, a card, a substitution), it puts a message on the Kafka topic match.events.
Four Kafka consumers in your NestJS app each read that message:

One updates Redis cache with the new score
One writes the event to Postgres permanently
One refreshes the materialized views (standings, top scorers)
One publishes the event via Redis Pub/Sub to push it to browsers

Why not just have the Poller call these four things directly? Because Kafka decouples them. If the Postgres write is slow, it doesn't slow down the Redis cache update. If consumer #4 crashes, consumers #1-3 still work. If you want to add a fifth consumer later (like push notifications), you add it without touching any existing code. And if your system restarts, Kafka remembers which messages each consumer has already processed — nothing is lost.
Why Redpanda and not Apache Kafka?
Redpanda speaks the exact same protocol as Apache Kafka. Your code uses KafkaJS (a Kafka client library) — it doesn't know whether it's talking to Apache Kafka or Redpanda. They're interchangeable.
The difference is purely operational. Apache Kafka requires Java, a JVM with memory tuning, and a separate coordination service (ZooKeeper or KRaft). Running it locally means 3+ Docker containers and complex configuration. Redpanda is a single binary, single container, starts in seconds, uses less memory.
Same protocol, same client library, same code, dramatically simpler to run. In the article you say "uses the Kafka protocol" and nobody cares which broker implementation is behind it — just like nobody cares if your app talks to MySQL via MariaDB or Oracle's MySQL.
Kafka locally vs production
Locally: A Docker container running Redpanda on your laptop. Your app connects to localhost:9092. No authentication, plain TCP, free.
Production: Redpanda Cloud (Serverless). Same Redpanda, but they run it for you on their servers. Your app connects to a cloud URL with username/password authentication and SSL encryption. This is the one external service you need because Render doesn't offer anything Kafka-compatible.
Your code doesn't change. The only difference is the connection string and whether authentication is enabled — handled by environment variables.
