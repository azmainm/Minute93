**Engineering Standards**

Version 1.0 — 4th March 2026

Created by Azmain Morshed, Lead Software Engineer

# **Purpose**

These standards exist to ensure code quality, maintainability, and reliability, regardless of whether code is written by a human or generated by AI.

AI-generated code that passes functional testing does not automatically meet our engineering bar. These rules ensure that every line of code committed is clean, structured, well-documented, and production-ready.

# **Tech Stack** (most commonly used \- may change)

* **Backend:** NestJS (TypeScript)

* **Frontend:** Next.js (TypeScript/React)

All naming conventions, file structures, and patterns must follow the conventions of the respective framework being used.

# **1\. Code Organization & Modularity**

* **Single Responsibility Principle —** every function and module does one thing and does it well.

* **Co-locate related functions —** group related utility functions in shared files and import them wherever needed. Maximize code reusability and minimize duplication.

* **Framework naming conventions —** follow the standard file naming patterns of the framework in use. For NestJS, this means files like user.controller.ts, user.service.ts, user.module.ts, etc. For Next.js, this means folder-based routing with page.tsx, layout.tsx, loading.tsx, etc. Do not invent custom conventions.

* **Max file length —** no file should exceed 300–400 lines without a strong justification. If it does, refactor into smaller modules.

* **Barrel exports —** use index files for clean import paths where appropriate.

* **Separation of concerns —** business logic, data access, presentation, and configuration must never be tangled together. Keep them in clearly separated layers.

# **2\. Error Handling & Resilience**

* **Mandatory try/catch —** wrap all async operations, API calls, database queries, and file I/O in try/catch blocks.

* **Centralized error handling —** use a shared error handler or middleware. Do not handle errors ad-hoc in every file.

* **Meaningful error messages —** every catch block must log context about what failed and why. Never use empty catches like catch(e) {}.

* **No silent catches —** catch(e) {} with no logging or handling is strictly forbidden. Every error must be at minimum logged.

* **Graceful degradation —** partial failures should not crash the entire system. Handle failures at the boundary where they occur.

* **Retry logic —** implement retry with exponential backoff for calls to external services.

* **Input validation —** validate inputs at every system boundary: API endpoints, form handlers, webhook receivers.

# **3\. Naming & Readability**

* **Descriptive names —** no single-letter variable names outside of loops. No abbreviations unless universally understood.

* **Boolean prefixes —** boolean variables must be prefixed with is, has, should, or can (e.g., isActive, hasPermission).

* **Functions as verbs —** function names describe actions: getUserById, not userData.

* **Constants —** use UPPER\_SNAKE\_CASE for constants. Extract them to dedicated config or constants files.

* **No magic numbers or strings —** every hardcoded value must be named and centralized.

# **4\. API & Data Layer Standards**

* **Consistent response structure —** all API responses must follow a uniform shape, e.g., { success, data, error, message }.

* **Correct HTTP status codes —** use the appropriate status codes (200, 201, 400, 401, 403, 404, 500). Never return 200 for errors.

* **Schema validation —** validate all request and response payloads with schemas (Zod, Joi, class-validator, etc.).

* **Pagination, filtering, sorting —** handle these consistently across all list endpoints.

* **Parameterized queries —** all database queries must be parameterized. Never use string interpolation in queries.

* **Transactions —** use database transactions for any operation that spans multiple writes.

# **5\. Logging & Observability**

* **Structured logging —** use severity levels: info, warn, error, debug. Use a proper logger, not console.log.

* **No console.log in production —** all production logging must go through the configured logger. console.log statements must be removed before merging.

* **Log at meaningful points —** log incoming requests, outgoing calls to external services, errors, and important state transitions.

* **No sensitive data in logs —** never log tokens, passwords, API keys, or personally identifiable information.

* **Correlation IDs —** pass request/correlation IDs through the call chain for end-to-end traceability.

# **6\. Configuration & Environment**

* **Nothing hardcoded —** all secrets, API keys, URLs, and environment-specific values must live in .env files (or .env.local for frontend). Import them as needed.

* **Fail fast on missing config —** if a required environment variable is missing at startup, the application must fail immediately with a clear error message rather than failing silently later.

# **7\. Testing**

* **Test after every feature —** after developing a feature, run tests to verify it works end-to-end. Only move on to the next task once the current feature passes testing.

* **Validate API inputs —** test that API endpoints handle invalid, missing, and edge-case inputs correctly.

* **No test interdependence —** no test should depend on another test’s state or execution order.

* **Mock external services —** never call real external APIs in tests. Mock all third-party service calls.

# **8\. Git & PR Standards**

* **Conventional commits —** use prefixed commit messages: feat:, fix:, refactor:, chore:, docs:, test:, etc.

* **Cohesive commits —** group related changes into a single commit. If a task has 10 subtasks and 3 are similar, commit those 3 together. If all 10 are unrelated, make 10 separate commits. Commits should be logically cohesive.

* **PR descriptions —** every PR must include: what changed, why it changed, how to test it, and any risks or known issues.

* **Human review required —** no PR is merged without at least one human code review. This is non-negotiable, especially for AI-generated code.

* **Branch naming —** use the format type/description, e.g., feat/user-authentication, fix/payment-timeout, refactor/api-response-handler.

# **9\. Branching Workflow**

For any new ticket, task, feature, bug fix, or bulk work:

1. Pull the latest from main.

2. Check out a new branch following the branch naming convention.

3. Commit in cohesive groups as per the commit standards above.

4. Run the build command *(npm run build* or equivalent) **before handing off.** Verify the build succeeds with no errors.

5. Push, open a PR with a complete description, and request a human review.

# **10\. Documentation**

## **10.1 README**

* **One README per codebase —** every repository must have a single, well-maintained README.md.

* **Keep it updated —** whenever a feature is added, a dependency changes, or setup steps change, the README must be updated in the same PR.

* **JSDoc —** use JSDoc (or equivalent) comments on all exported functions and classes.

* **API documentation —** if feasible, auto-generate API docs from schemas (e.g., Swagger/OpenAPI). This is recommended but not mandatory if tooling doesn’t support it.

## **10.2 CLAUDE.md**

At the start of any project, if no CLAUDE.md file exists, generate one. This file guides AI coding agents and must follow these principles:

* **Under 150 lines total —** keep it concise.

* **Universally applicable —** focus only on information relevant to the entire codebase, not edge cases.

* **Progressive disclosure —** instead of dumping all instructions into one file, create a brief index pointing to other markdown files in .claude/docs/ for specialized topics.

* **File:line references —** point to specific file and line references instead of pasting code snippets.

* **Assume linters for style —** do not include formatting rules that a linter can enforce.

CLAUDE.md must be structured as:

6. Project overview

7. Tech stack

8. Key directories and their purposes

9. Essential build and test commands

10. List of additional documentation files Claude should check when relevant

CLAUDE.md must be updated whenever significant changes are made to the codebase structure, tooling, or conventions.

## **10.3 Feature Documentation**

When a new feature is developed that warrants documentation (architecture decisions, integration details, etc.), proper documentation should be created. However, always check with the developer first whether documentation is actually needed before writing it — not every change requires a doc.

# **11\. Working with AI Coding Agents**

When using AI coding tools like Claude Code or Cursor for development work, follow this workflow to maximize code quality and minimize review overhead.

## **11.1 Before Starting a Session**

* **Prepare a clean requirements list —** before prompting the AI, write out a clear, organized list of all features, bug fixes, or tasks required for this session of work. Do not give vague or scattered instructions.

* **Flag if an implementation plan is needed —** if the task involves a major feature, architectural decisions, or complex logic, explicitly state in your prompt that you want an implementation plan first. For example: “Before writing any code, outline the implementation plan for the following requirements.”

* **Provide context —** if there are external files, API docs, design specs, or reference materials that would help the AI understand the task, mention them or attach them. If the relevant context already lives in the codebase, you can ask the AI to analyze the codebase first, or point it to specific filenames that are particularly important.

## **11.2 Plan Mode (Major Features Only)**

For new major features or tasks that require architectural planning:

* **Use plan mode —** if using Claude Code, activate plan mode so the AI outlines its approach before writing any code. This lets you review the proposed architecture, file structure, and implementation strategy against these engineering standards before a single line is written.

* **When to use plan mode —** this is required only when the prompt explicitly requests an implementation plan or when the task involves significant new architecture. It is not required for minor bug fixes, small tweaks, or straightforward tasks.

* **Review the plan —** a developer must review and approve the plan before the AI proceeds to code. Catch architectural issues early, not in PR review.

## **11.3 During the Session**

* **One logical task at a time —** let the AI complete and verify one task before moving to the next. Do not overload a single prompt with unrelated work.

* **Verify builds —** after the AI completes its work, always run the build command to confirm nothing is broken.

* **Review before committing —** never blindly commit AI-generated code. Review it against these standards, check for unnecessary complexity, and ensure it follows the patterns already established in the codebase.

# **Enforcement**

AI tools like Claude Code and Cursor must be configured (via CLAUDE.md, rules files, or plan mode prompts) to follow these standards before writing any code.

Code that does not meet these standards will be flagged in review and must be corrected before merging.
