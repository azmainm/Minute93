# API Standards

Standards for all NestJS API endpoints in Minute93.

## Response Structure

All API responses follow this uniform shape:

```typescript
interface ApiResponse<T> {
  success: boolean;
  data: T | null;
  error: string | null;
  message: string;
}
```

**Success example:**
```json
{ "success": true, "data": { "id": 1, "name": "Arsenal" }, "error": null, "message": "Team retrieved successfully" }
```

**Error example:**
```json
{ "success": false, "data": null, "error": "NOT_FOUND", "message": "Team with ID 999 not found" }
```

Never return raw data without wrapping it. Never return `200` for errors.

## HTTP Status Codes

| Code | When to use |
|------|-------------|
| 200 | Successful GET, successful PUT/PATCH |
| 201 | Successful POST that creates a resource |
| 204 | Successful DELETE (no body) |
| 400 | Validation error, malformed request |
| 401 | Missing or invalid authentication |
| 403 | Authenticated but not authorized (e.g., non-admin hitting /admin/*) |
| 404 | Resource not found |
| 409 | Conflict (e.g., duplicate email on signup) |
| 429 | Rate limited |
| 500 | Unhandled server error |

## Validation

- Use `class-validator` decorators on all DTOs.
- Register a global `ValidationPipe` with `whitelist: true` and `forbidNonWhitelisted: true`.
- Validate at the controller layer — services receive already-validated data.
- Return `400` with specific field errors for validation failures.

```typescript
// Example DTO
class CreateUserDto {
  @IsEmail()
  email: string;

  @IsString()
  @MinLength(8)
  password: string;
}
```

## Pagination

All list endpoints support pagination with consistent query parameters:

```
GET /matches?page=1&limit=20&sort=kickoff_at&order=desc
```

**Paginated response shape:**
```typescript
interface PaginatedResponse<T> {
  success: true;
  data: {
    items: T[];
    total: number;
    page: number;
    limit: number;
    totalPages: number;
  };
  error: null;
  message: string;
}
```

**Defaults:** `page=1`, `limit=20`, `order=desc`. Max `limit=100`.

## Filtering

List endpoints accept query parameters for filtering:

```
GET /matches?status=live&league_id=2
GET /matches?date=2026-06-15
```

Only allow filtering on indexed columns. Reject unknown filter parameters with `400`.

## Error Handling

- All errors go through the centralized `HttpExceptionFilter`.
- Never throw raw Error objects — use NestJS `HttpException` subclasses (`NotFoundException`, `BadRequestException`, etc.).
- Log every error with context: what was attempted, the input that caused it, and the stack trace.
- External service errors (API-Football, Redis, Kafka) must be caught, logged, and returned as `500` with a generic message — never expose internal details.

## Authentication & Authorization

- Protected routes use `@UseGuards(JwtAuthGuard)`.
- Admin routes use `@UseGuards(JwtAuthGuard, AdminGuard)`.
- Public routes (match listing, search, standings) have no guard.
- The SSE endpoint (`/matches/:id/events`) is public — no auth required for live updates.

## Correlation IDs

- Every request receives a `X-Request-ID` (generated by middleware if not present).
- This ID is passed through all service calls and logged with every log entry.
- Returned in the response header for client-side debugging.
